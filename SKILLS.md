## 1. The core idea (architectural inversion)

The developer writes **ordinary React + inline Tailwind** using a `dot.*`
namespace. Those components render **exactly as written** — `dot.panel` → `<div>`,
`dot.text` → `<p>`/heading/etc. There is **no separate styling file**.

Two artifacts derive from that authored JSX:

1. **Schema file** (`dotui build`) — a static, committed JSON description of every
   addressable dot/panel: structure, element tags, the developer's baseline
   classes, content (static vs dynamic), `required`/`description` metadata, and a
   palette. **The committed file is the source of truth** (regenerated by `dotui build`).
   It is *also* snapshotted into the DB additively (`schemas` table, deduped by content
   hash — §7) so overlays can be read against the exact schema they targeted; the DB is
   never authoritative over it. This is what an LLM reads to understand the UI.

2. **Style overlay** — a map keyed by dot id: `{ className?, style? }` per dot.
   A generator produces this (mock fallback or OpenAI in the demo). It is **merged**
   into the developer's real elements via `tailwind-merge` (override wins), so the
   overlay *augments* the live tree — it never replaces it.

```
  *.panel.tsx                  dotui build               .dotui/schema.json
  (dot.* JSX, authoring) ───►  extract + merge   ───►   (committed, constant)
        │                                                      │
        │ Babel plugin stamps __dotId                          │ read by LLM (M3)
        ▼                                                      ▼
  live React elements  ◄──── tailwind-merge ◄──── Overlay { dotId: {className,style} }
  (dot.* render real HTML)      (override wins)      persisted in libSQL (versioned)
```

The **id is the bridge**: the *same* structural id is computed at build time
(into the schema) and stamped onto the live element at runtime, so an overlay
maps back onto exactly the right element. See §3.

Behavior (onClick, onChange, hover, refs, aria/data, etc.) is **developer-owned**
and passes straight through `dot.*` — the overlay only touches className/style.

## 3. The id scheme (critical — read before touching the compiler)

An id is the element's **structural path** through the source:

```
<scope>/<ancestorSegments...>/<kind>#<siblingIndex>
```

- **scope** = file base name minus `.panel.tsx` (e.g. `profile.panel.tsx` → `profile`). `compiler/src/ids.ts:scopeFromFile`.
- **segment** = `<kind>#<index>`, where index is per-kind, per-parent (0-based). `makeSegment`.
- Examples (from the live demo schema):
  - `profile/panel#0` — the root panel of profile.panel.tsx
  - `profile/panel#0/panel#0/text#0` — first text inside the nested panel
  - `profile/panel#0/button#0` — the button directly in the root panel

**One shared walk guarantees consistency:** `compiler/src/walk.ts` exports
`findRootDots(nodes)` and `assignDotIds(roots, scope) → Map<JSXElement, string>`.
Both `dotui build` (extraction) **and** the Babel runtime plugin call this exact
function, so build-time ids === runtime stamped ids. **Never fork this logic.**

- Ids are **auto-generated only** — developers never supply ids.
- Within a file, ids can't collide (structural). Across files, the only way to
  collide is two files sharing a base name (same scope) → `mergeSchemas` reports
  it and `dotui build` **fails (exit 1)**. (`compiler/src/merge.ts`)
- Stamping is **idempotent** (`plugin.ts:stampId` skips if `__dotId` present).

## 4. `@dotui/core` — the contract

No runtime deps except `zod`. Everyone imports from here.

**`src/kinds.ts`** — closed vocabulary (single source of truth for compiler + runtime):
- `PANEL_KIND = 'panel'`
- `DOT_KINDS = ['text','button','input','image','badge']` → `type DotKind`
- `DEFAULT_ELEMENT` — kind → default HTML tag (`panel:div, text:p, button:button, input:input, image:img, badge:span`)
- `ALLOWED_ELEMENTS` — kinds whose `type` prop selects a tag. **Only `text`** today: `['p','span','label','h1'..'h6','strong','em','small']`. For other kinds `type` is a normal DOM attr (e.g. `dot.input type="email"`).
- `VOID_KINDS = ['input','image']` — no children/content.
- `isDotKind(value)` guard.

**`src/guardrail.ts`** — `ALLOWED_FAMILIES` (17 Tailwind families: text-size,
font-weight, text-color, bg-color, padding, margin, gap, rounded, border,
border-color, display, flex, grid, align, justify, width, height). This is the
vocabulary advertised to the LLM. **Class-by-class validation is NOT built yet** (M3).

**`src/types/schema.ts`** — the schema-file types:
- `ContentPart = {kind:'static', value} | {kind:'dynamic', expr}`
- `DotNode = { id, kind, element, className, dynamicClassName, required, description?, content }`
- `PanelNode = { id, kind:'panel', element, className, dynamicClassName, description?, children: string[] }`
- `Palette = { seededFromAuthor: string[], families: string[] }`
- `SchemaFile = { version, panels: Record<id,PanelNode>, dots: Record<id,DotNode>, palette }`

**`src/types/overlay.ts`** — React-free so server/store can import it:
- `StyleOverride = { className?: string; style?: Record<string,string|number>; hidden?: boolean }`
  — `hidden` makes the dot render nothing (a hidden panel drops its whole subtree). Visibility lives
  on the overlay, not the schema; it's how prompts/personas reveal or remove authored elements.
- `Overlay = Record<string, StyleOverride>` (keyed by dot id)

**`src/schema.ts`** — Zod mirrors with `.strict()`, `SCHEMA_VERSION = 1`. Drives
`dotui build` validation. Mirror any type change here too.

## 5. `@dotui/compiler` — authoring → schema + runtime ids

Deps: `@babel/parser`, `@babel/types`, `fast-glob`, `@dotui/core`.

- **`src/ast.ts`** — focused readers over Babel JSX nodes (no traversal):
  `dotKindOf`, `isDotElement`, `elementName`, `readClassName` (static strings &
  zero-expr template literals only; else `{value:'', dynamic:true}`),
  `readStringProp` (for `type`, `description`), `readBooleanProp` (bare
  `required` → true, `={false}` → false, absent → false), `readContent`
  (JSXText → static, JSXExpressionContainer → dynamic w/ raw source slice).
- **`src/ids.ts`** — `makeSegment`, `makeId`, `scopeFromFile`.
- **`src/walk.ts`** — `findRootDots`, `assignDotIds` (THE shared id source, §3).
- **`src/extract.ts`** — `extractFile(filePath, code) → { schema, warnings }`.
  Parses TS+JSX, assigns ids, builds nodes. Panels always use `DEFAULT_ELEMENT.panel`
  (ignore `type`). Dots resolve element via `resolveElement` (only ALLOWED_ELEMENTS
  kinds treat `type` as tag). Seeds palette from author's used classes. Warns on
  non-dot elements inside panels (they render but aren't addressable) and on
  nested elements inside text dots.
- **`src/merge.ts`** — `mergeSchemas(inputs) → { schema, conflicts }`. Enforces
  globally-unique ids across the combined dot+panel namespace.
- **`src/plugin.ts`** — `dotuiBabelPlugin()` → Babel plugin `dotui-stamp-ids`;
  Program visitor stamps `__dotId="<id>"` on every dot via the shared walk.
- **`src/cli.ts`** — `run(argv)` for `dotui build <glob...> [--out <path>]`.
  Defaults: glob `**/*.panel.tsx`, out `.dotui/schema.json`. Extracts each file,
  merges, **fails on conflicts**, validates with `schemaFileSchema`, writes JSON.
  Exports `run()` with no import side effect (testable).
- **`src/bin.ts`** — the executable entry; sets `process.exitCode = run(process.argv.slice(2))`.
- **`src/index.ts`** exports: `extractFile`, `mergeSchemas`, `dotuiBabelPlugin`,
  `assignDotIds`, `findRootDots`, `makeId`, `makeSegment`, `scopeFromFile`.
- `tsup.config.ts` builds two entries: `src/index.ts` (lib) and `src/bin.ts` (CLI, `dist/bin.cjs`).

## 6. `@dotui/elements` — the `dot.*` components

Deps: `react`, `tailwind-merge`, `@dotui/core`.

- **`src/dot.tsx`** — `export const dot = { panel, text, button, badge, input, image }`.
  - Props type: `ComponentPropsWithRef<T> & DotMeta` where `DotMeta = { __dotId?, description?, required? }`. **ComponentPropsWith*Ref*** (not WithoutRef) so `ref` forwards.
  - Each component **destructures and strips** `__dotId, description, required` (not real DOM attrs), then spreads `...rest` (all events/aria/data/ref) and the merged className/style.
  - `useMerged(dotId, className, style)` reads the overlay for that id and returns `{ className: twMerge(author, override), style: {...author, ...override} }` — **override wins**.
  - `dot.text` accepts `type` ∈ TextTag union to pick its tag (defaults `p`).
- **`src/overlay.tsx`** — `DotOverlayProvider({value, children})` (React context of the `Overlay`) and `useOverlay(dotId?)`.
- **`src/index.ts`** exports `dot`, `DotOverlayProvider`, `useOverlay`, and the `Overlay`/`StyleOverride` types (re-exported from core via `src/types.ts`).
- Tests (happy-dom + @testing-library/react): `tests/interactions.test.tsx` (events, disabled/type/aria/data passthrough, hover/focus class preservation, ref forwarding); `tests/merge.test.tsx` (overlay merge, override-wins, no `__dotId` leak, metadata strip, tag from `type`).

## 7. `@dotui/store` — versioned overlay persistence

Deps: `drizzle-orm`, `@libsql/client`, `@dotui/core`.

- **`src/schema.ts`** — two Drizzle tables. **`overlays`**: `id` (autoinc PK), `panelId`,
  `scope` (default `'default'`), `appSchemaVersion`, `prompt`, `overlayJson`,
  `version`, `parentVersion`, `isCurrent` (bool), `createdAt` (default
  `CURRENT_TIMESTAMP`). **`schemas`** (build-schema snapshots): `id` (autoinc PK),
  `formatVersion`, `hash` (content hash, **UNIQUE** → dedup), `schemaJson`, `isCurrent`,
  `createdAt`. Ships raw DDL for both (`OVERLAYS_DDL`/`OVERLAYS_INDEX_DDL`,
  `SCHEMAS_DDL`/`SCHEMAS_INDEX_DDL`) so a fresh DB bootstraps without drizzle-kit.
  **Model:** every generation is a new versioned overlay row; `is_current=true` marks the
  live row per (scope, panelId). Schema snapshots follow the same is_current model but
  are deduped by content hash (re-saving identical content is a no-op).
- **`src/hash.ts`** — `stableHash(value)`: SHA-256 over canonical JSON (recursively
  sorted keys) so equal schema content always hashes equal. The schema dedup key.
- **`src/client.ts`** — `createDb(url) → { db, client }` (libSQL; `file:` local or
  `libsql://` hosted, same code), `ensureSchema(client)` runs the DDL.
- **`src/types.ts`** — `SaveOverlayArgs`, `OverlayRecord`, and the **`OverlayStore`
  interface** (the seam for swapping libSQL → Postgres later):
  - `save(args) → OverlayRecord` — new version, made current.
  - `current(scope?) → Overlay` — merge of every panel's current row (what renders).
  - `history(panelId, scope?) → OverlayRecord[]` — one panel, newest first.
  - `list(scope?) → OverlayRecord[]` — every version across all panels, newest first (for devtools).
  - `setCurrent(panelId, version, scope?)` — roll back.
  - `saveSchema(schema) → SchemaRecord` — snapshot the build schema, made current; **idempotent** (dedup by content hash).
  - `currentSchema() → SchemaRecord | null` — the live snapshot.
  - `listSchemas() → SchemaRecord[]` — every snapshot, newest first (for devtools).
- **`src/store.ts`** — `LibsqlOverlayStore implements OverlayStore`,
  `createOverlayStore(url)`. `save` runs in a transaction (read max version, unset
  current, insert v+1). Lazy `ensureSchema` on first call.
- Tests (`tests/store.test.ts`): versioning/current, cross-panel merge, list
  newest-first, rollback. **Uses a temp FILE per test** (`:memory:` can't share a
  connection across libSQL transactions).

## 7.5 `@dotui/devtools` — the reusable inspector

Deps: `react` (peer), `@dotui/core` (types only). **No dependency on `@dotui/store`** —
it consumes the wire JSON, so it never drags libSQL/drizzle into the client.

- **`<DotuiDevtools schema endpoint? pollMs? />`** — a floating button → TanStack-style
  panel with two tabs. Read-only. `schema: SchemaFile` is passed by the consumer
  (they import their own `.dotui/schema.json`); `endpoint` (default `/api/overlay`)
  must return `{ current: Overlay, records: OverlayRecordView[] }`; `pollMs` (default 2000).
- **Schema tab** (`schema-view.tsx`) — the committed build artifact: version, palette
  (families + author-seeded classes), every panel (element/classes/description/children)
  and every dot (kind→element, classes, content static/dynamic, `required`, description).
- **Overlays tab** (`overlays-view.tsx`) — the `overlays` DB table: a column legend +
  rows grouped by panel (newest first), each expandable to show every column value **and**
  the stored overlay JSON (per-dot className/style).
- **`src/types.ts`** — `OverlayRecordView` (the wire mirror of store's `OverlayRecord`,
  declared here so the package stays store-free — the JSON endpoint is the seam),
  `DevtoolsData`, `DotuiDevtoolsProps`, `DevTab`.
- **`src/use-overlay-records.ts`** — `useOverlayRecords(endpoint, pollMs, active)` polls
  while open, swallows transient errors, ignores in-flight responses after unmount.
- **`src/index.ts`** exports `DotuiDevtools` + re-exports `SchemaFile` (so consumers can
  type the `schema` prop without a direct `@dotui/core` dep) + the view-data types.
- Tailwind-styled — a consuming app must add `packages/devtools/src/**` to its Tailwind
  `content` globs (the demo does) so the inspector's classes are generated.
- Tests (`tests/views.test.tsx`, happy-dom): SchemaView renders panels/dots/slots/palette;
  OverlaysView lists rows + live badge + empty state. **3 tests.**

## 7.6 `@dotui/guardrail` — keep generated classes in-rail

Deps: `@dotui/core`. Pure logic, no React.
- **`classifyClass(token)`** — a single Tailwind utility → `AllowedFamily | null`. Strips variant
  prefixes (`hover:`, `md:`) and a leading negative; distinguishes `text-lg` (size) from `text-red-500`
  (color), `border-2` (border) from `border-slate-200` (border-color), bare `flex`/`grid` (display).
- **`validateClassName(s)`** → `{ className, dropped }`; **`validateOverlay(overlay)`** →
  `{ overlay, dropped }` — strips out-of-rail classes from every override, **keeps `hidden`/`style`**,
  reports dropped classes per dot id. The safety net when a real LLM emits junk.

## 7.7 `@dotui/llm` — prompt → overlay (mock + OpenAI engines)

Deps: `@dotui/core`, `@dotui/guardrail`. No React.
- **`panelSlice(schema, panelId)`** → `{ panels, dots }` — the subtree rooted at a panel (the focused
  context for one generation).
- **`Generator`** type = `(req) => Overlay | Promise<Overlay>` — the seam. `GenerateRequest =
  { prompt, panelId, panels, dots, palette?, current? }`.
- **`mockGenerator`** — deterministic keyword rules over the slice: bigger/smaller, dark, contrast,
  a colour word, rounded, spacious; plus **hide/show** by matching words after a verb against each
  node's text (kind + description + content). Emits only guardrail-friendly classes + `hidden`.
- **`openaiGenerator({ apiKey, model?, client? })`** — server-side Responses API generator. It forces
  one `emit_overlay` function call, parses strict `{ patches: [{ id, className, hidden }] }`
  arguments into an `Overlay` (the parser also tolerates keyed `overlay` objects), retries once on
  invalid JSON or guardrail-dropped classes, and is testable with an injected client.
- **`generatePanelOverlay(req, generator)`** → `{ overlay, dropped }` = run generator then guardrail.
  **The one call the UI/server route makes** — never apply raw model output directly.

## 7.8 `@dotui/prompt` — the in-product ✨ edit layer

Deps: `react` (peer), `@dotui/core`, `@dotui/elements`. Reusable, like devtools.
- **`<DotuiPromptProvider generate initialOverlay>`** — owns the live overlay, renders
  `DotOverlayProvider` so every dot picks it up, and shows the floating **✨ Edit UI** control
  (bottom-left). `generate: (args:{prompt,panelId,current?}) => Promise<{overlay, dropped?}>` is the
  app-supplied bridge. **Apply-only — no persistence** (prompting ≠ creating a version).
- **`<Promptable panelId>`** — wraps a panel; in edit mode overlays a corner ✨ that opens an inline
  per-panel prompt. Registers its id so the **global** prompt box can target every panel.
- `mergeOverlay(base, patch)` stacks successive prompts (classes accumulate, tailwind-merge resolves;
  `hidden`/`style` from the patch win). `usePrompt()` exposes the context.
- **Runtime classes need a Tailwind safelist** in the consuming app (the generator emits classes not
  present in source) — see the demo `tailwind.config.ts`.

## 11. Known frictions / gotchas (so you don't re-hit them)

- **esbuild isolatedModules:** keep value vs `type` re-exports split in every `index.ts`.
- **Babel plugin in tests:** traverse with explicit braces — a dangling `else` previously skipped non-array object children. The walk in `walk.ts` is correct; mirror it if you write new traversal.
- **`@babel/runtime` pin:** demo pins `@babel/runtime: 7.26.0` — a custom `babel.config.js` disables SWC, `next/babel` needs the regenerator subpath that 7.27+ dropped.
- **libSQL + webpack:** `@libsql/client` and `libsql` must be **direct deps of the demo** AND in `serverExternalPackages` (native module can't be bundled).
- **libSQL `:memory:`** doesn't share a connection across transactions → tests use a temp file.
- **`type` prop overload:** tag-selector only for kinds in `ALLOWED_ELEMENTS` (today just `text`); otherwise it's a passthrough DOM attr.
- **`required`/`description`** are dotUI metadata, captured into the schema and stripped by the components — they are not forwarded to the DOM.

styles every rendered instance. Both apps **also embed the prompt layer**: each panel is wrapped in
`<Promptable>` inside a `<DotuiPromptProvider>` with a server-only `app/api/generate` route (OpenAI when
`OPENAI_API_KEY` is set in `apps/<app>/.env.local`, else mock). Persona = base overlay (`initialOverlay`),
prompts stack on top; switching persona remounts the provider (`key={persona}`). Remaining work:

1. **Persist a prompted look** — an explicit "create version" action (apply ≠ create): take the
   current overlay, `overlayStore.save(...)`. Optionally let the user re-apply a saved version
   (`setCurrent`) from the devtools/history.
2. **Guardrail palette pass (optional)** — also constrain colours to the schema's `seededFromAuthor`
   palette, not just the families.

Dynamic `{param}` slots are already captured as opaque `content.dynamic` parts the generator preserves.
